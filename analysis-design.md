# Analysis and Design Document
## ระบบร้านขายหนังสือ

---

## 1. Problem Statement | คำบรรยายปัญหา

ร้านขายหนังสือแบบขายส่งส่วนใหญ่ยังคงใช้การจดบันทึกหรือระบบที่ไม่เป็นมาตรฐาน ทำให้เกิดปัญหาต่อไปนี้:

- ❌ การตรวจสอบสต็อกสินค้าต้องทำด้วยตนเองโดยไปดูสินค้าจริง
- ❌ ข้อมูลการสั่งซื้อบ่อยครั้งถูกบันทึกผิดพลาด
- ❌ การติดตามการชำระเงินยากและเกิดการสูญหายข้อมูล
- ❌ ลูกค้าไม่สามารถตรวจสอบสถานะคำสั่งซื้อแบบรีอลไทม์ได้
- ❌ ไม่มีรายงานสรุปยอดขายโดยอัตโนมัติ
- ❌ ใช้เวลามากในการประมวลผลข้อมูล

---

## 2. Pain Points

### 2.1 ฝั่งลูกค้า (Customer Pain Points)

| ปัญหา | ความรุนแรง | ผลกระทบ |
| ----- | --------- | --------|
| ค้นหาหนังสือได้ยาก | สูง | สูญเสียลูกค้าที่อยากซื้อ |
| ไม่ทราบจำนวนสินค้าคงเหลือ | สูง | สั่งซื้อแล้วไม่มีสินค้า |
| ตรวจสอบสถานะคำสั่งซื้อไม่สะดวก | สูง | ลูกค้ากังวลและไม่พึงพอใจ |
| การสั่งซื้อจำนวนมากใช้เวลานาน | กลาง | ลดประสิทธิภาพ |
| ไม่มีประวัติการซื้อ | กลาง | ยากต่อการค้นหาคำสั่งซื้อเก่า |

### 2.2 ฝั่งร้านค้า/พนักงาน (Staff Pain Points)

| ปัญหา | ความรุนแรง | ผลกระทบ |
| ----- | --------- | --------|
| ตรวจสอบสต็อกด้วยตนเอง | สูง | ใช้เวลา ข้อมูลอาจผิด |
| ข้อผิดพลาดในการบันทึก | สูง | นำไปสู่การจัดส่งผิดจำนวน |
| ติดตามการชำระเงินยาก | สูง | สูญเสียรายได้ |
| ไม่มีรายงานอัตโนมัติ | กลาง | ต้องประมวลผลด้วยตนเอง |
| การจัดส่งสับสน | กลาง | ลูกค้าไม่พึงพอใจ |

### 2.3 ฝั่งผู้บริหาร (Management Pain Points)

| ปัญหา | ความรุนแรง | ผลกระทบ |
| ----- | --------- | --------|
| ไม่มีการวิเคราะห์ข้อมูลยอดขาย | สูง | ตัดสินใจยากและไม่แม่นยำ |
| ไม่มีระบบควบคุม | สูง | อาจเกิดการทุจริต |
| ความไม่เสมอภาค | กลาง | ลูกค้าอาจไม่พึงพอใจ |

## 3. Functional Requirements | ความต้องการเชิงหน้าที่

### 3.1 Customer (ลูกค้า)

| FR ID | ความต้องการ | คำอธิบาย |
| ----- | --------- | --------|
| **FR-01** | User Registration | สมัครสมาชิกด้วยอีเมล ชื่อผู้ใช้ รหัสผ่าน และข้อมูลส่วนตัว |
| **FR-02** | User Authentication | เข้าสู่ระบบด้วยชื่อผู้ใช้และรหัสผ่าน |
| **FR-03** | Book Search | ค้นหาหนังสือตามชื่อ ผู้แต่ง หมวดหมู่ |
| **FR-04** | Book Details | ดูรายละเอียดหนังสือ (ชื่อ ราคา ผู้แต่ง สำนักพิมพ์ จำนวนสินค้า) |
| **FR-05** | Place Order | สั่งซื้อหนังสือพร้อมเลือกปริมาณ |
| **FR-06** | Payment | ชำระเงินผ่านหลายช่องทาง (เงินสด โอนธนาคาร บัตรเครดิต) |
| **FR-07** | Order Status | ตรวจสอบสถานะคำสั่งซื้อ |
| **FR-08** | Order History | ดูประวัติการซื้อ |

### 3.2 Admin (ผู้ดูแลระบบ)

| FR ID | ความต้องการ | คำอธิบาย |
| ----- | --------- | --------|
| **FR-10** | Payment Verification | ตรวจสอบและยืนยันการชำระเงิน |
| **FR-11** | Update Shipping Status | อัปเดตสถานะการจัดส่ง |
| **FR-12** | Inventory Check | ตรวจสอบสต็อกสินค้า |
| **FR-13** | Stock Report | ดูรายงานสินค้าขาด |
| **FR-14** | Book Management | เพิ่ม แก้ไข ลบข้อมูลหนังสือ |
| **FR-15** | Category Management | จัดการหมวดหมู่และหมวดย่อย |
| **FR-16** | Sales Report | ดูรายงานยอดขายและการวิเคราะห์ |

## 4. Non-Functional Requirements | ความต้องการที่ไม่ใช่หน้าที่

| NFR ID | หมวดหมู่ | ความต้องการ |
| ------ | -------- | ----------- |
| **NFR-01** | Performance | ระบบต้องแสดงผลหน้าเว็บและผลการค้นหาภายใน 3 วินาที ภายใต้สภาวะการใช้งานปกติ |
| **NFR-02** | Usability | หน้าจอต้องใช้งานง่าย เข้าใจได้โดยไม่ต้องมีคู่มือ  |
| **NFR-03** | Security | ข้อมูลผู้ใช้และรหัสผ่านต้องถูกจัดเก็บอย่างปลอดภัย (เข้ารหัส/Hash) และมีการตรวจสอบสิทธิ์การเข้าถึงในแต่ละบทบาท (Customer / Admin) |
| **NFR-04** | Reliability | ระบบต้องสามารถทำงานได้อย่างต่อเนื่อง มีการสำรองข้อมูลอย่างสม่ำเสมอเพื่อป้องกันข้อมูลสูญหาย |
| **NFR-05** | Availability | ระบบควรพร้อมให้บริการอย่างน้อย 95% ของเวลาทำการ ไม่รวมช่วงเวลาบำรุงรักษาที่แจ้งล่วงหน้า |
| **NFR-06** | Scalability | โครงสร้างระบบต้องรองรับการเพิ่มจำนวนสินค้าและผู้ใช้งานในอนาคตได้โดยไม่ต้องออกแบบระบบใหม่ทั้งหมด |
| **NFR-07** | Maintainability | โค้ดต้องเป็นระเบียบ มีการควบคุมเวอร์ชันผ่าน Git/GitHub เพื่อให้ง่ายต่อการแก้ไขและต่อยอดในอนาคต |
| **NFR-08** | Compatibility | ระบบต้องรองรับการใช้งานผ่านเว็บเบราว์เซอร์หลัก เช่น Chrome, Firefox, Edge และ Safari |
| **NFR-09** | Data Integrity | ข้อมูลสต็อกสินค้าและคำสั่งซื้อต้องมีความถูกต้องตรงกันระหว่างฝั่งลูกค้าและฝั่งหลังบ้านตลอดเวลา |
